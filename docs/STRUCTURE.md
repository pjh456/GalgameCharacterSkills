# 项目结构与开发路线

本项目正在进行二次重构，本文档用于记录项目的重构架构设计与路线图

## 项目架构

### 基建层

该部分模块不存在上游模块依赖，均对单一任务领域进行封装，并对高级任务提供接口。

#### fs 文件系统模块

项目涉及到大量文件读写操作，因此考虑将其集中封装，以实现代码的复用。

模块负责：
- 直接读写文件获取原始信息
- 从特定文件解析出 Python 可用格式，如 `.json`、`.yaml` 和 `.env` 等
- 检查文件和目录的存在性

模块不负责：
- 对保存内容做除格式以外的任何修改
- 检查内容的合法性

#### net 网络请求模块

全项目围绕 API 调用展开，故将基础的异步网络请求与接口调用集中封装。

模块负责：
- 封装网络请求内容，避免直接对外暴露细节
- 向 OpenAI 兼容接口请求发送数据，并获取其返回值
- 实现阻塞与非阻塞式的网络请求，供上游不同需求使用
- 在请求失败时提供详细的报错信息

模块不负责：
- 对请求体进行详细解析
- 判断请求体的合法性
- 调度请求任务

#### log 日志模块

为提高项目的 Debug 效率，避免静默失败问题，封装日志模块进行统一输出。

模块负责：
- 在日志文件夹分层记录不同级别的详细日志
- 为上游异常状态打印提供查询接口
- 写入日志文件

模块不负责：
- 将日志信息输出到控制台或前端

注：由于日志的特殊性，设计时使之与 fs 模块独立，以便于日志文件的管理。

#### conf 配置模块

目前的项目需要全局配置与任务配置分离，以支持 API 参数的持久化和断点重传任务。

数据类定义、执行上下文、静态配置单例，在设计上都可以包含在这一模块之内。

模块负责：
- 提供为下游封装的数据类，如任务与请求封装
- 提供任务执行过程的上下文，收集参数信息统一处理
- 在本地读取和保存全局通用参数，如 API Key、BaseURL、切片 Token 阈值等
- 在本地读取和保存任务参数，包括指定的角色、自定义指令等

模块不负责：
- 直接读写文件

注：fs 层将会为读写配置信息提供接口，但在逻辑上 conf 模块仍然被下游多个模块依赖，故将其放在基建层，但并不会对基建层的独立性产生破坏。

### 功能层

通过调用基建层暴露接口，实现逻辑初步封装，并分别实现对应需求功能。

本层模块仅依赖基建层，且同层间不存在相互依赖关系。

#### llm AI 调用模块

由网络层获取到的请求不应由外部直接发起调用，因此使用 llm 对单个 AI 调用请求封装，并提供统一调用接口。

简而言之，llm 模块负责“接收一次已经构造完成的模型请求，并返回调用结果”。

模块负责：
- 接收上游构造好的模型请求参数并发起调用
- 处理单次调用中的请求异常与返回异常
- 提供阻塞与非阻塞调用接口
- 返回结构统一的调用结果与错误信息

模块不负责：
- 读取任务配置或执行上下文
- 调度多个 AI 调用任务
- 管理任务执行进度与恢复状态
- 对获取的结果进行进一步处理与保存

#### executor 任务执行模块

这一模块实际上包含了任务的完整执行过程，可能涉及到多个 llm 实例的调用、执行上下文读取、任务数据的获取和进度持久化等。

模块负责：
- 执行单一任务并保存进度
- 提供详细的任务执行进度信息
- 从指定任务恢复并继续执行

模块不负责：
- 直接读写全局配置与任务配置文件
- 处理与执行多任务

### 逻辑层

使用功能层封装的接口，专注于逻辑实现，组织起不同模块的功能。

特点是封装内涉及到上游多个模块的引入，以组织逻辑。

#### engine 执行引擎模块

该模块负责组织功能层模块，对外提供统一的调用入口。

模块负责：
- 根据输入参数选择并构造对应任务
- 调用合适的 executor 执行任务
- 组织多个功能模块之间的协作流程
- 对执行结果进行统一收集、包装与返回
- 处理上层接口所需的整体调用逻辑

模块不负责：
- 直接发起底层网络请求
- 实现单次 AI 调用细节
- 直接读写配置与文件
- 承担具体任务内部的逐步执行逻辑

#### api 接口模块

提供网络请求接口供前端使用，实际上内嵌一个 engine 用于实际的逻辑执行。
