# 项目结构与开发路线

本项目正在进行二次重构，本文档用于记录项目的重构架构设计与路线图

## 项目架构

### 核心层

该层定义全项目共享的基础抽象、通用结果对象和公共模型，不承载具体逻辑和执行。

核心层的目标是为上层提供统一、稳定、低耦合的公共定义，并作为依赖关系的最底层。

#### core 核心模块

模块负责：
- 定义全项目共享的通用结果封装
- 定义公共基础类型与模型
- 定义与具体业务无关的路径、资源定位或工作区结构模型
- 为上层模块提供统一的低层抽象基础

模块不负责：
- 保存任务业务状态
- 承担具体文件读写、日志写入或网络请求逻辑
- 处理任务调度、执行流程与接口组织
- 依赖其他业务模块

### 基建层

该层建立在核心层公共定义之上，均对单一任务领域进行封装，并对高级任务提供接口。

#### fs 文件系统模块

项目涉及到大量文件读写操作，因此考虑将其集中封装，以实现代码的复用。

模块负责：
- 基于 core 层提供的公共结果对象与路径模型，封装文件系统读写能力
- 直接读写文件获取原始信息
- 从特定文件解析出 Python 可用格式，如 `.json`、`.yaml` 和 `.env` 等
- 检查文件和目录的存在性

模块不负责：
- 对保存内容做除格式以外的任何修改
- 检查内容的合法性

#### net 网络请求模块

全项目围绕 API 调用展开，故将基础的异步网络请求与接口调用集中封装。

模块负责：
- 封装网络请求内容，避免直接对外暴露细节
- 向 OpenAI 兼容接口请求发送数据，并获取其返回值
- 实现阻塞与非阻塞式的网络请求，供上游不同需求使用
- 在请求失败时提供详细的报错信息

模块不负责：
- 对请求体进行详细解析
- 判断请求体的合法性
- 调度请求任务

#### log 日志模块

为提高项目的 Debug 效率，避免静默失败问题，封装日志模块进行统一输出。

模块负责：
- 基于 core 层提供的公共结果对象与基础模型，封装日志记录与查询能力
- 在日志文件夹分层记录不同级别的详细日志
- 提供面向调试与排错的日志查询接口
- 写入并读取日志文件

模块不负责：
- 保存进程级日志配置状态
- 替代任务状态、检查点等业务状态存储
- 承担任务执行流程控制

设计约定：
- 与 fs 模块保持职责独立
- 日志配置由上游模块显式传入
- 日志记录可按任务隔离

#### conf 配置模块

目前的项目需要全局配置与任务配置分离，以支持 API 参数的持久化和断点重传任务。

该模块负责定义项目运行所需的配置模型、任务模型与任务状态模型。

模块负责：
- 提供为下游封装的数据类，如任务配置、请求封装等
- 提供任务执行过程的状态数据结构，收集参数信息统一处理
- 定义运行配置与任务配置的结构，提供统一的数据模型基础

模块不负责：
- 保存进程级全局单例状态
- 直接读写文件
- 隐式向下游模块分发配置

为避免并发任务之间互相污染运行状态，项目中的运行配置不通过模块级全局变量共享。

配置传递约定如下：
- 运行配置由最上层入口统一构造
- 需要配置的模块在初始化时显式接收配置对象
- 任务执行过程中的可变状态仅保存在任务状态对象中
- 下游模块不主动读取隐式全局配置
- 不同任务之间的运行时状态应当相互隔离

注：conf 模块在子模块 module 中集中了需要注入上下文的其他模块配置。

### 功能层

通过调用基建层暴露接口，实现逻辑初步封装，并分别实现对应需求功能。

本层模块仅依赖基建层，且同层间不存在相互依赖关系。

#### llm AI 调用模块

由网络层获取到的请求不应由外部直接发起调用，因此使用 llm 对单个 AI 调用请求封装，并提供统一调用接口。

简而言之，llm 模块负责“接收一次已经构造完成的模型请求，并返回调用结果”。

模块负责：
- 接收上游构造好的模型请求参数并发起调用
- 处理单次调用中的请求异常与返回异常
- 提供阻塞与非阻塞调用接口
- 返回结构统一的调用结果与错误信息

模块不负责：
- 读取任务配置或执行上下文
- 调度多个 AI 调用任务
- 管理任务执行进度与恢复状态
- 对获取的结果进行进一步处理与保存

#### executor 任务执行模块

这一模块实际上包含了任务的完整执行过程，可能涉及到多个 llm 实例的调用、执行上下文读取、任务数据的获取和进度持久化等。

模块负责：
- 执行单一任务并保存进度
- 提供详细的任务执行进度信息
- 从指定任务恢复并继续执行
- 在单任务范围内持有并使用执行所需依赖，如运行配置、日志实例、调用模块等

模块不负责：
- 直接读写全局配置与任务配置文件
- 处理与执行多任务间的总体调度
- 通过隐式全局状态获取运行参数

### 逻辑层

使用功能层封装的接口，专注于逻辑实现，组织起不同模块的功能。

特点是封装内涉及到上游多个模块的引入，以组织逻辑。

#### engine 执行引擎模块

该模块负责组织功能层模块，对外提供统一的调用入口。

模块负责：
- 根据输入参数选择并构造对应任务
- 构造任务执行所需的运行配置与依赖实例
- 调用合适的 executor 执行任务
- 组织多个功能模块之间的协作流程
- 对执行结果进行统一收集、包装与返回
- 处理上层接口所需的整体调用逻辑

模块不负责：
- 直接发起底层网络请求
- 实现单次 AI 调用细节
- 直接读写配置与文件
- 承担具体任务内部的逐步执行逻辑

#### api 接口模块

提供网络请求接口供前端使用，实际上内嵌一个 engine 用于实际的逻辑执行。

模块负责：
- 接收外部请求并解析输入参数
- 加载本次调用所需的运行配置
- 调用 engine 完成实际逻辑执行
- 将执行结果转换为上层接口需要的返回格式
