# 项目结构与开发路线

本项目正在进行二次重构，本文档用于记录项目的重构架构设计与路线图

## 项目架构

### 基建层

该部分模块不存在上游模块依赖，均对单一任务领域进行封装，并对高级任务提供接口。

#### fs 文件系统模块

项目涉及到大量文件读写操作，因此考虑将其集中封装，以实现代码的复用。

模块负责：
- 直接读写文件获取原始信息
- 从特定文件解析出 Python 可用格式，如 `.json`、`.yaml` 和 `.env` 等
- 检查文件和目录的存在性

模块不负责：
- 对保存内容做除格式以外的任何修改
- 检查内容的合法性

#### net 网络请求模块

全项目围绕 API 调用展开，故将基础的异步网络请求与接口调用集中封装。

模块负责：
- 封装网络请求内容，避免直接对外暴露细节
- 向 OpenAI 兼容接口请求发送数据，并获取其返回值
- 实现阻塞与非阻塞式的网络请求，供上游不同需求使用
- 在请求失败时提供详细的报错信息

模块不负责：
- 对请求体进行详细解析
- 判断请求体的合法性
- 调度请求任务

#### log 日志模块

为提高项目的 Debug 效率，避免静默失败问题，封装日志模块进行统一输出。

模块负责：
- 在日志文件夹分层记录不同级别的详细日志
- 为上游异常状态打印提供查询接口
- 写入日志文件

模块不负责：
- 将日志信息输出到控制台或前端

注：由于日志的特殊性，设计时使之与 fs 模块独立，以便于日志文件的管理。

#### conf 配置模块

目前的项目需要全局配置与任务配置分离，以支持 API 参数的持久化和断点重传任务。

模块负责：
- 在本地读取和保存全局通用参数，如 API Key、BaseURL、切片 Token 阈值等
- 在本地读取和保存任务参数，包括指定的角色、自定义指令等
- 作为任务执行过程的上下文，收集参数信息

模块不负责：
- 直接读写文件

注：fs 层将会为读写配置信息提供接口，但在逻辑上 conf 模块仍然被下游多个模块依赖，故将其放在基建层，但并不会对基建层的独立性产生破坏。

### 功能层

通过调用基建层暴露接口，实现逻辑初步封装，并分别实现对应需求功能。

本层模块仅依赖基建层，且同层间不存在相互依赖关系。

#### task 任务模块

无论是断点重传还是正常处理，都涉及到总结原始文本、归纳总结切片两个步骤。

将其分别封装为任务，是符合直觉和逻辑的，也便于断点重传的实现。

该模块将会直接依赖于 conf 模块，以获取详细的任务信息。

模块负责：
- 提供不同任务的数据类封装
- 通过上游接口检查任务执行状态
- 对外提供任务执行进度
- 自动检查并恢复续传

模块不负责：
- 对任务的结果进行额外处理
- 手动处理配任务配置

#### llm AI 调用模块

由网络层获取到的请求不应由外部直接发起调用，使用 llm 对单个 AI 调用请求封装，并提供更高层级接口。

换而言之，task 和 llm 分别负责了 AI 请求的数据和逻辑。

模块负责：
- 读取一个任务并对应开始 AI 请求
- 校验并处理报错的异常请求
- 异步执行以允许多并发请求
- 提供详细的执行信息

模块不负责：
- 调度多个 AI 调用任务
- 对获取的结果进行进一步处理与保存

### 逻辑层

使用功能层封装的接口，专注于逻辑实现，组织起不同模块的功能。

特点是封装内涉及到上游多个模块的引入，以组织逻辑。

#### engine 执行引擎模块

这一模块串联起关于数据处理的全逻辑，并提供了完整的执行流程，以此对外进行调用。

#### api 接口模块

提供网络请求接口供前端使用，实际上内嵌一个 engine 用于实际的逻辑执行。
